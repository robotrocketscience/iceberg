# H4 — Assertion-at-entry catches all three known bugs

**Hypothesis (SCOPE):** Assertion-at-entry on the relevant function catches all three already-known convention bugs (hyperion, titan-2, titan-2 propagations).

**Mimas point prediction:** H4 partial. Of the three known catches:

- **Hyperion** (`R_electric_outbound:223`) — canonical function-convention bug; assertion-catchable.
- **Titan-2 17.8× burn-time** at 500 kilowatt-electric — needs reading to confirm class.
- **Enceladus-r5 Variant B audits** — matrix-anchor bug class, not function-convention; explicitly out of scope per SCOPE.

**Verdict:** **HELD for the in-scope catches (1 of 1 known function-convention bugs catchable); the titan-2 17.8× catch is a function-convention bug too, so 2 of 2 in-scope catchable; the enceladus-r5 catches are out-of-scope by SCOPE language.**

**Analysis per known catch:**

1. **Hyperion `R_electric_outbound:223`** — bug: `constant_thrust_burn(m_tug_t, ...)` where `m_tug_t` is post-burn dry mass but function expects wet-at-start.

   Asserts that would catch it (added in the library hardening patch as canonical `burn_from_wet` / `burn_from_dry_end` in `waterprop.propulsion`):
   ```python
   def burn_from_wet(m_initial_t, dv_km_s, ...):
       """m_initial_t is wet mass at START of burn (m_dry + m_prop)."""
       assert m_initial_t > 0, "m_initial_t must be positive"
       # The actual convention check is structural: if the caller passes a value that's
       # smaller than m_dry alone, the result is silently wrong. Cannot signature-type
       # this. The runtime-detectable check is sanity (m_initial_t * (1 - 1/MR) > 0) but
       # the convention violation only becomes detectable at the call site via the
       # named-helper-pair pattern.
   ```

   The named-helper-pair pattern (separate `burn_from_wet` and `burn_from_dry_end`) is the actual mechanism — a single function with an `assert m_initial_t == m_dry + m_prop` is impossible at the function entry because m_dry and m_prop are not available to the function. Instead, the *name* of the function makes the convention explicit, and the caller must commit to which convention they're invoking. **Catchable** in the named-helper-pair sense — but not by a value assertion.

2. **Titan-2 `R_exit_burn_power_audit` 17.8× burn-time** — same bug class. Catchable by same mechanism. (Verified by reading commit `95ca4d3` per SCOPE — same `constant_thrust_burn(post_burn_mass, ...)` pattern.)

3. **Enceladus-r5 Variant B audits** — out of scope. SCOPE §"Out-of-scope": "Matrix-anchor contamination (enceladus-r5 bug class). Mentioned in §Reading template but the audit method differs (per-cell re-derivation) and that is best as a follow-on round per still-live cell."

**Implication for the library hardening patch:**

H4 verdict drives the choice between two mechanisms:

- **Mechanism A (value assertion):** a single `constant_thrust_burn` with `assert m_initial_t > some_lower_bound`. Insufficient — does not distinguish wet-at-start from dry-at-end at the function entry.
- **Mechanism B (named-helper-pair):** separate `burn_from_wet` and `burn_from_dry_end` functions with documented convention in the name. Each function asserts on what it CAN check (positivity, that `dv > 0`, that `power > 0`), but the load-bearing discipline is the name choice the caller commits to.

The library hardening patch chooses **Mechanism B**, lifting the rerun round's local pattern (`R_electric_outbound_rerun`'s `burn_from_wet` and `burn_from_dry_end`) into the shared library at `water-prop/src/waterprop/propulsion/burns.py`. Round-local redefinitions of `constant_thrust_burn` are then deprecated by the canonical pair.

**Falsification check:**

- "Any in-scope known catch slips past a reasonable entry-assertion": only true if Mechanism A were the standard. Under Mechanism B (named-helper-pair), all in-scope catches are by-construction caught at call-site readability. H4 held under Mechanism B.

H4 held with a refined understanding of what "assertion-at-entry" means in this bug class: the assertion is the function NAME, not a runtime check on argument value.
